# AI 텍스트 감정 분석 서비스 프로젝트 보고서

## 1. 문제 정의

### 1.1 대상 문제
실시간으로 입력된 텍스트의 감정(긍정/부정/중립)을 분석하여 사용자의 감정 상태를 파악하는 AI 서비스 개발

### 1.2 사용자 페르소나
- **주 사용자**: 개발자, 데이터 분석가
- **사용 목적**:
  - 고객 리뷰 감정 분석
  - 소셜 미디어 모니터링
  - 챗봇 응답 개선을 위한 감정 파악

### 1.3 기대 효과
- 텍스트 데이터의 감정을 자동으로 분류하여 수동 분석 시간 단축
- RESTful API를 통한 쉬운 통합
- Docker 기반 배포로 일관된 실행 환경 제공

### 1.4 제약 사항
- **시간**: 5일 이내 완료
- **자원**: 로컬 개발 환경, GPU 없이 CPU만 사용
- **모델**: 가벼운 사전 훈련된 모델 활용 (BERT 기반)
- **언어**: 한국어 및 영어 지원

## 2. 기능 범위 (MVP)

### 2.1 입력
- HTTP POST 요청을 통한 텍스트 입력
- JSON 형태: `{"text": "분석할 텍스트"}`

### 2.2 출력
- JSON 형태의 감정 분석 결과
- 형태: `{"sentiment": "positive|negative|neutral", "confidence": 0.95}`

### 2.3 품질 지표
- **응답 시간**: 1초 이내
- **정확도**: 80% 이상 (테스트 데이터셋 기준)
- **가용성**: 99% 업타임

## 3. 시스템 아키텍처

### 3.1 전체 구조
```
Client Request → FastAPI → AI Model → Response
                     ↓
                 Docker Container
                     ↓
                Health Check
```

### 3.2 구성 요소
- **Web Framework**: FastAPI
- **AI Model**: Hugging Face Transformers (BERT)
- **Container**: Docker
- **Documentation**: OpenAPI/Swagger

## 5. 결과

### 5.1 구현 완료 사항
- ✅ **Docker 환경 구축**: 멀티스테이지 빌드, 보안 설정 완료
- ✅ **FastAPI 서비스**: 4개 엔드포인트 구현 (/health, /predict, /model/*)
- ✅ **AI 모델 통합**: Hugging Face Transformers 기반 감정분석
- ✅ **테스트 코드**: 20개 이상의 단위 테스트 작성
- ✅ **자동화**: Makefile로 30개 명령어 자동화
- ✅ **문서화**: 포괄적인 API 문서 및 아키텍처 문서

### 5.2 기술적 성과
- **코드 품질**: Pydantic 검증, 구조화된 로깅, 에러 핸들링
- **보안**: 비루트 사용자, 입력 검증, 환경변수 관리
- **확장성**: 모듈화된 구조, 의존성 주입 패턴
- **모니터링**: 헬스체크, 처리시간 측정, 상태 추적

- **감정 분석**: 텍스트 → positive/negative/neutral
- **신뢰도 점수**: 0-1 범위의 확률값
- **처리 시간**: 밀리초 단위 성능 측정
- **OpenAPI 문서**: Swagger UI 제공

## 6. 토론 및 분석

### 6.1 설계 결정 분석
**장점**:
- **Docker 사용**: 일관된 배포 환경, 의존성 격리
- **FastAPI 선택**: 자동 문서화, 타입 힌트, 비동기 지원
- **Pydantic 검증**: 런타임 타입 안전성, 자동 문서화
- **모듈화**: 유지보수성, 테스트 용이성

**트레이드오프**:
- **CPU vs GPU**: GPU 미사용으로 처리속도 제한, 하지만 배포 단순화
- **사전훈련 모델**: 도메인 특화 부족, 하지만 개발시간 단축
- **메모리 사용**: 모델 상주로 메모리 사용량 증가

### 6.2 성능 예상
- **응답 시간**: CPU 환경에서 1-3초 예상
- **동시 사용자**: 제한적 (단일 프로세스)
- **확장 방안**: 로드밸런서, 모델 서빙 최적화 필요

### 6.3 아키텍처 강점
- **관심사 분리**: API/Model/Utils 계층 분리
- **설정 관리**: 환경변수 기반 구성
- **에러 처리**: 계층별 예외 처리
- **테스트**: 모킹을 통한 단위 테스트

## 7. 한계점 및 윤리적 고려사항
- **한계점**:
  - CPU 환경에서의 처리 속도 제한
  - 사전 훈련된 모델의 도메인 특화 정확도 제한
- **윤리적 고려**:
  - 개인정보 보호: 입력 텍스트는 저장하지 않음
  - 편향성: 모델의 학습 데이터 편향 가능성 인지
